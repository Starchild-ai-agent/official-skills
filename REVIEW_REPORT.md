## 📋 一、项目概览

### 为什么做这件事？

Starchild 的核心竞争力在于 **Skills**——一套让 AI Agent 具备真实能力的工具模块。但随着 skill 数量增长到 **18 个**、代码量达到 **6.6 万行**，我们发现：

1. **代码质量参差不齐** — 早期 skill 缺少错误处理、返回值未过滤
2. **小模型适配不足** — Gemini Flash / GPT-4o Mini 上下文有限，臃肿的返回值直接吃掉窗口
3. **缺乏量化标准** — "好不好"全靠人肉 review，没有可复现的度量

### 我们做了什么？

构建了一套 **完整的自动化质量评估 + 强化学习优化闭环**，让 AI 自己审查自己的代码，并持续自我改进。

## 🏗️ 二、技术架构

### 评估矩阵（损失函数设计）

$$L_{total} = \omega_1 \cdot L_{task} + \omega_2 \cdot L_{efficiency} + \omega_3 \cdot L_{cost} + \omega_4 \cdot L_{density}$$

| 维度 | 含义 | 检测方式 | 权重 (小模型) |
|------|------|---------|-------------|
| **L_task** | 功能完整性 | 缺少错误处理 = 功能缺陷 | **ω₁ = 10** |
| **L_efficiency** | 调用效率 | try/except 覆盖率 | **ω₂ = 3** |
| **L_cost** | Token 消耗 | 代码行数 / SKILL.md 大小 | **ω₃ = 1** |
| **L_density** | 上下文密度 | 原始JSON返回、冗余字段 | **ω₄ = 5** |

> **设计哲学**: 权重反映了小模型场景的优先级——功能正确最重要 (10)，其次是不浪费宝贵的上下文窗口 (5)，再是少走弯路 (3)，最后是省钱 (1)。

### 优化引擎

```
┌─────────────────────────────────────────────────┐
│                 Optimization Loop                │
│                                                  │
│   ┌──────────┐    ┌──────────┐    ┌──────────┐  │
│   │  Scan    │───→│  Patch   │───→│  Verify  │  │
│   │ (Score)  │    │ (Atomic) │    │ (Δ<0?)   │  │
│   └──────────┘    └──────────┘    └─────┬────┘  │
│        ↑                                │        │
│        │          ┌──────────┐    ┌─────▼────┐  │
│        └──────────│  Report  │←───│  Keep /  │  │
│                   │ (Track)  │    │  Revert  │  │
│                   └──────────┘    └──────────┘  │
└─────────────────────────────────────────────────┘
```

**关键机制：**

| 机制 | 说明 |
|------|------|
| **ε-greedy 探索** | 80% 时间优化最差 skill, 20% 随机探索防止局部最优 |
| **原子补丁** | 每次只改一个地方, 精确衡量 ΔLoss |
| **自动回滚** | ΔLoss ≥ 0 → 自动恢复, 保护代码不退化 |
| **递减收益曲线** | L = n/(n+k), 早期修复收益大, 防止过拟合单一维度 |
| **半小时调度** | 每 30 分钟自动执行一轮, 持续进化 |

## 🤖 五、小模型适配策略

### 为什么特别关注小模型？

| 模型层级 | 代表 | 上下文窗口 | 安全阈值* |
|----------|------|-----------|----------|
| 🟢 Large | GPT-4o, Claude Opus | 128k | 16,000 |
| 🟡 Medium | GPT-4o Mini, Claude Sonnet | 64k | 8,000 |
| 🔴 Small | Gemini Flash, Claude Haiku | 32k | **6,000** |

> *安全阈值 = 留给 skill 输出的预算。超过此值, 模型可能截断或幻觉。

### 适配权重调优

小模型的权重配置直接影响优化方向：

```python
# 大模型: 均衡分布, 上下文不是瓶颈
LARGE  = (ω_task=10, ω_eff=2.0, ω_cost=1, ω_density=1)

# 中模型: 开始关注密度
MEDIUM = (ω_task=10, ω_eff=2.5, ω_cost=1, ω_density=3)

# 小模型: 密度成为第二优先级 (仅次于功能正确性)
SMALL  = (ω_task=10, ω_eff=3.0, ω_cost=1, ω_density=5)
```

## 💡 九、方法论亮点

### 1. 借鉴强化学习的工程实践

这不是传统的 code review。我们把 **代码质量优化** 当作一个 **RL 问题**来解：

- **状态空间 (S)** = 所有 skill 的当前代码  
- **动作空间 (A)** = 原子补丁 (try/except, 字段过滤, 截断, 内联)  
- **奖励函数 (R)** = -ΔLoss (损失下降 = 正奖励)  
- **策略 (π)** = ε-greedy + 优先修复最差 skill  

### 2. 为什么递减收益 > 硬上限

早期设计用 `min(n/k, 1.0)` 作为损失函数, 导致修复一个 bug 无法被感知 (因为还有65个, 还是满分1.0)。改为 `n/(n+k)` 后:

```
66 findings: L = 66/(66+5) = 0.930
65 findings: L = 65/(65+5) = 0.929  → Δ = -0.001  ← 可以被检测到!
```

这让优化器能正确识别"方向对了, 继续走"。

### 3. 小模型适配 ≠ 简单压缩

我们的密度损失不是要求"输出越短越好"——而是要求：
- ✅ 只返回有用的字段 (过滤 raw JSON)
- ✅ SKILL.md 精简到核心 (< 6000 tokens)
- ✅ 错误信息简洁明确 (不倒冗余 traceback)

这些改进对**所有**模型都有益, 只是对小模型**至关重要**。
